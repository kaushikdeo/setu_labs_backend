# Module D: Engineer Visit Management

## Overview

Handles scheduling engineer visits to customer sites, assigning equipment tasks per visit, capturing execution fields during validation/calibration, and enforcing master instrument expiry checks.

---

## Data Models

### Visit (`setulab_backend/src/modules/visit/visit.model.ts`)

| Field | Type | Notes |
|---|---|---|
| code | string | Autogenerated `VISIT-<timestamp>` |
| customerId | ObjectId | Ref: Customer |

## Status Transitions

```
Visit:     SCHEDULED → IN_PROGRESS → COMPLETED
                    ↘ CANCELLED

VisitTask: PENDING → IN_PROGRESS → COMPLETED
                              ↘ FAILED
```

- Visit auto-transitions to `IN_PROGRESS` when first task is started.
- Visit auto-transitions to `COMPLETED` when all tasks reach `completed`.

---

## Instrument Expiry Check

When `POST /api/visits/:id/tasks/:taskId/start` is called:

1. Check if task has `instrumentId` assigned.
2. If yes, fetch the `MasterInstrument` record.
3. If `calibrationDueDate < now` or `status !== ACTIVE` → throw `AppError(403, ...)`.
4. Auto-update instrument status to `expired`.
5. If valid → set task status to `in_progress`.

## API Endpoints

```
GET    /api/visits                            — list (filter: status, type, assignedEngineerId)
POST   /api/visits                            — create (super_admin, validation_head)
GET    /api/visits/:id                        — detail + tasks
PATCH  /api/visits/:id                        — update (super_admin, validation_head)
GET    /api/visits/:id/tasks                  — list tasks
POST   /api/visits/:id/tasks                  — add task (authenticated)
PATCH  /api/visits/:id/tasks/:taskId          — update task fields
POST   /api/visits/:id/tasks/:taskId/start    — start task (expiry check)
POST   /api/visits/:id/tasks/:taskId/complete — complete task + fill execution fields
```

### Create flow (`/visits/add`)

Single scrollable page with a sticky 6-step stepper. All sections are visible at once; clicking a step scrolls to that section.

| Step | Section | Data |
|---|---|---|
| 1 | Service details | `type`, `scheduledDate` |
| 2 | Customer & site | `customerId`, `siteId` |
| 3 | Area / equipment | One or more equipment rows |
| 4 | Master instruments | Instruments for active area |
| 5 | Tests to perform | Test types for active area |
| 6 | Assign engineer | `assignedEngineerId`, optional `notes` |

Environmental conditions are **not** captured at create time. See [TEST_ENVIRONMENTAL_CONDITIONS.md](../../../System_Design_Docs/TEST_ENVIRONMENTAL_CONDITIONS.md).

Submit sequence:
1. `POST /api/visits` with visit payload
2. For each equipment row: `POST /api/visits/:id/tasks` with `{ equipmentId, plannedTests }`
3. Redirect to `/visits/:id`

Edit mode shows steps 1, 2, 6 only; equipment/tasks remain on the detail page.
